## 一、Web Search — Agent 自主联网搜索

### 功能说明

**默认已关闭。** 开启后，CoderAgent 和 ModelerAgent 可以调用 `search_web` 工具搜索互联网，获取真实数据（如统计数据、文献信息、最新研究结果）注入到建模和代码中。

搜索结果会被结构化为 `DataEvidence`（含 unit、time_range、region 等字段），并注入到 Agent 的 prompt 中。

### 启用方式

唯一需要做的是配置 Tavily API Key：

1. 注册 [Tavily](https://tavily.com) 账号并获取 API Key（免费 1000 次/月）
2. 在 `backend/.env.dev` 中设置：
   ```env
   TAVILY_API_KEY=tvly-xxxxxxxxxxxxxxxx
   ```

未配置 API Key 时，Web Search 自动跳过（日志 warning），不影响主流程。

### 自定义配置（可选）

```env
SEARCH_ENABLED=true          # 默认已关闭，设为 true 可开启
SEARCH_CACHE_TTL=86400       # 搜索缓存过期时间（秒），默认 24 小时
```

### 工作原理

```
用户提交建模问题
    │
    ├─ CoordinatorAgent 解析题目
    ├─ ModelerAgent 制定方案（可调用 search_web 获取背景数据）
    └─ CoderAgent 编写代码（可调用 search_web 获取真实数据填充代码）
```

Agent 在 tool_call 中返回 `{"name": "search_web", "arguments": {"query": "...", "data_type": "statistics"}}`，系统自动调用 Tavily API 搜索，再由 LLM 提取结构化数据注入 prompt。

### 验证方法

1. 配置 API Key 后启动后端，提交一个需要真实数据的建模问题
2. 观察日志中是否出现 `search_web` 工具调用
3. 生成的代码中应包含真实数据（而非占位符）

## 二、RAG 知识库 — 本地知识检索

### 功能说明

**默认已关闭。** 开启后，从本地知识库中检索建模方法、代码模板、论文写作参考，注入到 Agent 的 prompt 中。基于 ChromaDB 向量数据库 + CrossEncoder Rerank。

三个 Agent 分别检索不同类型的知识：

| Agent | 检索 source_type | 用途 |
|-------|-----------------|------|
| ModelerAgent | paper, textbook | 建模方法 + 优秀论文方案 |
| CoderAgent | code | 代码模板 |
| WriterAgent | paper, problem | 论文写作模板 |

### 启用方式

RAG 默认已关闭。开启后需要构建知识库才能检索到内容，没有知识库时自动跳过，不影响主流程。

**第一步：准备知识库文件**

```
backend/data/knowledge/
    papers/       # 论文 PDF
    textbooks/    # 教材（Markdown/文本）
    code/         # 代码模板（.py/.jl/.r）
    problems/     # 优秀论文方案
```

将你的 PDF、Markdown、代码文件放入对应目录即可。

**第二步：构建索引**

```bash
cd backend
uv sync                                    # 安装依赖（含 chromadb、sentence-transformers 等）
python scripts/build_knowledge_base.py     # 构建 ChromaDB 索引
```

索引保存在 `data/chromadb/` 目录，支持增量重建（每次全量覆盖）。

### 自定义配置（可选）

```env
RAG_ENABLED=true                    # 默认已关闭，设为 true 可开启
RAG_DB_PATH=data/chromadb           # ChromaDB 存储路径
RAG_TOP_K=5                         # 返回结果数量
RAG_EMBEDDING_MODEL=BAAI/bge-m3     # Embedding 模型
RAG_RERANKER_MODEL=BAAI/bge-reranker-v2-m3  # Reranker 模型
```

### 工作原理

```
workflow.execute()
    │
    ├─ ModelerAgent 启动前 → 检索 paper/textbook 知识 → 注入 prompt
    ├─ CoderAgent 启动前 → 检索 code 知识 → 注入 prompt
    └─ WriterAgent 启动前 → 检索 paper/problem 知识 → 注入 prompt

Agent 也可主动调用 search_knowledge 工具检索
```

### 验证方法

1. 构建索引后启动后端，观察日志中是否出现 `知识检索完成: query=..., 找到 N 条`
2. 如果没有知识库文件，日志会提示 `ChromaDB 未初始化，跳过知识检索`，不影响流程

## 三、HIL 人机协作 — 关键节点审批

### 功能说明

**默认已开启。** 工作流在关键节点会暂停并发送审批消息到前端，等待用户做出决策。支持 6 种决策动作：

| 动作 | 行为 |
|------|------|
| **confirm** | 确认当前结果，继续下一步 |
| **edit** | 用用户修改后的内容继续 |
| **regenerate** | 用原始 prompt 重跑当前 Agent |
| **ask** | 暂停等待用户补充信息后继续 |
| **skip** | 跳过审核，直接继续 |
| **abort** | 清理资源，标记任务失败 |

无需任何配置，启动后即可在前端看到审批对话框。超时（默认 300 秒）自动 confirm 继续。

### 自定义配置（可选）

```env
HIL_ENABLED=true                    # 默认已开启，设为 false 可关闭
HIL_TIMEOUT=300                     # 审批超时时间（秒），超时自动 confirm
HIL_CHECKPOINTS={"problem_split":true,"model_selection":true,"code_review":false,"paper_review":true}
```

### 检查点说明

| 检查点 | 默认 | 说明 |
|--------|------|------|
| `problem_split` | 开 | CoordinatorAgent 拆题后暂停，审核子任务拆分是否合理 |
| `model_selection` | 开 | ModelerAgent 出方案后暂停，审核建模方案是否正确 |
| `code_review` | 关 | 每个子任务 CoderAgent 执行后暂停（按需开启，会频繁打断） |
| `paper_review` | 开 | WriterAgent 写完全文后暂停，审核论文质量 |

### 工作原理

```
workflow.execute()
    │
    ├─ CoordinatorAgent 解析题目
    │   └─ checkpoint: problem_split → 发送 ApprovalMessage → 等待决策
    ├─ ModelerAgent 制定方案
    │   └─ checkpoint: model_selection → 发送 ApprovalMessage → 等待决策
    ├─ CoderAgent × N 求解
    │   └─ checkpoint: code_review_{key}（默认关闭）
    └─ WriterAgent 撰写论文
        └─ checkpoint: paper_review → 发送 ApprovalMessage → 等待决策
```

决策通过双向 WebSocket 传递：前端 `sendDecision()` → 后端 `state_store.set()` → `checkpoint_manager.wait_for_update()` 返回。

### 前端使用

审批对话框（`ApprovalDialog.vue`）会自动弹出，显示：
- 当前结果内容（可编辑）
- 6 个决策按钮
- 倒计时进度条（超时自动 confirm）

## 四、Fallback Hand Off — 主模型故障自动切换

### 功能说明

当主 LLM 调用失败（超时、API 错误、返回无效 JSON）时，自动切换到备用模型重试。切换时**新建 Agent 实例**，避免脏状态污染。

未配置备用模型时，该功能自动跳过，仅使用主模型。

### 配置方式（可选）

按需为每个 Agent 配置备用模型，未配置的 Agent 不启用 Fallback：

```env
# ---- Coordinator Fallback ----
FALLBACK_COORDINATOR_API_KEY=sk-yyy
FALLBACK_COORDINATOR_MODEL=gpt-4o-mini
FALLBACK_COORDINATOR_BASE_URL=https://api.openai.com/v1
FALLBACK_COORDINATOR_MAX_TOKENS=4096

# ---- Modeler Fallback ----
FALLBACK_MODELER_API_KEY=sk-yyy
FALLBACK_MODELER_MODEL=gpt-4o-mini
FALLBACK_MODELER_BASE_URL=https://api.openai.com/v1
FALLBACK_MODELER_MAX_TOKENS=4096

# ---- Coder Fallback ----
FALLBACK_CODER_API_KEY=sk-yyy
FALLBACK_CODER_MODEL=gpt-4o-mini
FALLBACK_CODER_BASE_URL=https://api.openai.com/v1
FALLBACK_CODER_MAX_TOKENS=4096

# ---- Writer Fallback ----
FALLBACK_WRITER_API_KEY=sk-yyy
FALLBACK_WRITER_MODEL=gpt-4o-mini
FALLBACK_WRITER_BASE_URL=https://api.openai.com/v1
FALLBACK_WRITER_MAX_TOKENS=4096
```

> 仅当 `API_KEY` 和 `MODEL` 均配置时才创建 Fallback 实例。

### 有限重试配置

```env
# Coordinator JSON 解析最大重试次数（默认 5）
MAX_COORDINATOR_RETRIES=5

# Modeler JSON 解析最大重试次数（默认 5）
MAX_MODELER_RETRIES=5
```

Writer 的最大重试次数在代码中默认为 3 次，Coder 已有 `MAX_RETRIES` 配置。

### 工作原理

```
Agent.run(prompt)
    │
    ├─ 成功 → 返回结果
    │
    └─ 异常 → fallback_llm 存在？
        │
        ├─ 是 → 新建 Agent(task_id, fallback_llm) → 重试
        │       └─ 成功 → 返回结果
        │       └─ 失败 → 抛出异常
        │
        └─ 否 → 抛出原始异常
```

### 前端提示

Hand Off 触发时，前端会收到 SystemMessage：
- `协调者主模型失败(ValueError)，正在切换备用模型...`（warning）
- `代码手主模型失败，正在切换备用模型...`（warning）
- `写作手执行出错，正在重试(1/3)...`（warning）
- `写作手超过最大重试次数(3)，最后错误: XXX`（error）

## 五、Evaluator + Feedback Rerun — 质量评估与反馈重跑

### 功能说明

使用独立的便宜 LLM 评估 Agent 输出质量：
- **Shadow Mode**（默认）：只记录评分，不触发重跑
- **Feedback Rerun**：评估不通过时，将反馈注入 prompt 重跑 Agent

未配置评估器时，该功能自动跳过。

### 配置方式（可选）

```env
# ---- 评估器配置 ----
EVALUATOR_API_KEY=sk-xxx
EVALUATOR_MODEL=gpt-4o-mini
EVALUATOR_BASE_URL=https://api.openai.com/v1

# ---- Feedback Rerun 配置 ----
# 最大 Feedback 重跑轮数（默认 2）
MAX_FEEDBACK_ROUNDS=2

# 评估通过阈值，0-1（默认 0.6）
EVALUATION_THRESHOLD=0.6
```

> 未配置 `EVALUATOR_API_KEY` 和 `EVALUATOR_MODEL` 时，评估和 Feedback Rerun 均跳过。

### 工作原理

```
_run_writer_with_feedback(writer_agent, prompt, evaluator)
    │
    for round in 0..MAX_FEEDBACK_ROUNDS:
    │
    ├─ 1. 执行 Agent（含 Hand Off 保护）
    ├─ 2. 无评估器或最后一轮 → 直接返回结果
    ├─ 3. 评估通过（passed=True 或 score >= threshold）→ 返回结果
    └─ 4. 评估不通过 → 拼接反馈到 prompt → 新建 Agent → 继续循环
```

Feedback 注入方式：
```python
augmented = f"{original_prompt}\n\n【评估反馈（第{round}轮）】\n{eval_result.feedback}"
```

### 评估日志

前端会收到评估结果的 SystemMessage：
- `[评估] ques1 代码手评分: 0.85 (通过)`（info）
- `[评估] ques1 写作手评分: 0.40 (未通过)`（warning）
- `写作手 ques1 评估未通过(score=0.40)，注入反馈重跑...`（warning）

## 七、常见问题

### Q: 不配置任何新功能，系统行为会变吗？

**不变。** Web Search 和 RAG 默认已关闭，HIL 默认已开启。各功能均可通过环境变量独立开关：
- `SEARCH_ENABLED=false`（默认）→ 不加载搜索工具
- `RAG_ENABLED=false`（默认）→ 不加载知识检索工具
- `HIL_ENABLED=true`（默认）→ 关键节点暂停等待审批，超时自动放行
- Fallback / Evaluator 未配置 → 跳过对应功能

### Q: Web Search 搜索结果不准确怎么办？

Tavily 的搜索质量取决于 query 的构造。Agent 会自动生成搜索关键词，你可以在 prompt 中引导 Agent 搜索特定方向的数据。

### Q: HIL 审批超时了会怎样？

默认超时 300 秒后自动执行 `confirm`（确认当前结果继续）。可通过 `HIL_TIMEOUT` 调整。

### Q: Fallback 模型和主模型用同一个 API 可以吗？

可以，但建议用不同模型（如主模型用 gpt-4o，Fallback 用 gpt-4o-mini），这样主模型 API 故障时 Fallback 更可能成功。

### Q: Evaluator 评估器用什么模型好？

推荐用便宜的模型（如 gpt-4o-mini），因为评估只做单次打分，不需要复杂推理。

### Q: Feedback Rerun 最多跑几轮？

默认 2 轮（`MAX_FEEDBACK_ROUNDS=2`），加上首次执行共 3 轮。每轮会新建 Agent 实例，避免状态污染。

### Q: 如何关闭某个功能？

在 `backend/.env.dev` 中设置对应开关为 false 即可，例如 `HIL_ENABLED=false`。各功能独立开关，互不影响。

### Q: RAG 知识库没有知识库文件会报错吗？

**不会。** RAG 开启后，如果没有构建知识库（`data/chromadb/` 不存在），检索会自动跳过并返回空结果，不影响主流程。只需将文件放入 `data/knowledge/` 并运行 `python scripts/build_knowledge_base.py` 即可启用知识检索。
